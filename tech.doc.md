# ТЕХНИЧЕСКОЕ ЗАДАНИЕ (ИТОГОВАЯ ВЕРСИЯ)
## Telegram-бот для администрирования моргов (MorgueBot)

### 1. Общее описание
Бот управляет двумя моргами:
- Первомайская 13
- Мира 11

### 2. Пользователи и роли
- Роли: `админ`, `менеджер_морга1`, `менеджер_морга2`, `агент_морга1`, `агент_морга2`
- Пользователи хранятся в `users.json` (Telegram ID → роль)
- Бот определяет морг пользователя по роли

### 3. Хранение данных
- `morgue1.json` — данные морга Первомайская 13
- `morgue2.json` — данные морга Мира 11
- `users.json` — пользователи
- Формат: JSON, синхронизация через GitHub (вручную)
- Все JSON-файлы хранятся в папке `data/`

### 4. Смены

**Открытие:**
- Автоматически при первом нажатии «Добавить труп+» или любого действия, требующего смены
- Фиксируется `start_time`, `opened_by`

**Действия в смене:**
- Добавление трупа
- Удаление трупа (выезд в БСМЭ) — запись помечается `removed=true`, не влияет на финансы смены
- Оформление ритуального заказа (привязка к `shift_id`)

**Закрытие:**
- Кнопка «Закрыть смену» (менеджер своего морга / админ с выбором морга)
- Автоматически в 00:00 следующего дня, если не закрыта вручную:
  - Все трупы считаются оплаченными
  - Ритуальных заказов не было
- При закрытии — статистика сразу выводится в чат

### 5. Трупы

**При внесении:**
- Фамилия
- Категория (стандарт / нестандарт)
- Тип поступления (стационар / амбулаторно)

**При закрытии смены:**
- Бот показывает список фамилий
- Для каждого: статус оплаты (оплачен / не оплачен)
- Если не оплачен → обязательное поле «кто вывез (организация)»

**Удаление трупа:**
- Кнопка «Удалить труп» → выбор из списка текущей смены
- Причина: «БСМЭ» (или текст)
- Запись в истории, но не участвует в финансах смены
- Участвует в аналитике «кто вывез»

### 6. Финансы (доход/расход)

**Доход за тело (одинаково для обоих моргов):**
- Стандарт → 8 000 руб
- Нестандарт → 10 000 руб

**Морг «Первомайская 13»:**
- Расход (санитары):
  - Стандарт → 5 500 руб
  - Нестандарт → 8 000 руб

**Морг «Мира 11»:**
- Расход (санитары):
  - Стандарт → 6 500 руб
  - Нестандарт → 8 000 руб
- Доходной части у морга нет (прибыль = 0)

**Зарплата агента (добавляется в расход смены):**
- Рассчитывается **при закрытии смены**, после учёта оплаты трупов
- Если в смене были заказы на ритуальные услуги:
  - Бот запрашивает у менеджера (или админа) зарплату по каждому заказу:
    - `Похороны (Иванов) — зарплата агента: [ввод суммы]`
    - `Кремация (Петров) — зарплата агента: [ввод суммы]`
  - Менеджер вводит сумму вручную
- Все суммы суммируются → **общая зарплата агента за смену**
- Добавляется в **общий расход смены**
- Если заказов нет — зарплата не запрашивается

### 7. Статистика за смену (при закрытии)

Выводится сразу в чат:
```
Всего тел: X
Доход: XXXX руб
Расход (санитары): XXXX руб
Зарплата агента: XXXX руб
Общий расход: XXXX руб
Чистая прибыль: XXXX руб

Неоплаченные:
- Фамилия — Организация
```

### 8. Статистика (кнопка для админа)

- Доступна **только админу**
- Выбор: морг 1 / морг 2 / оба морга
- Показывает:
  - Итоговые цифры (всего тел, доход, расход, прибыль)
  - **Список вывезенных:** дата, фамилия, организация (кто вывез)

### 9. Отчёт за период (неделя / месяц / квартал)

- Доступен админу и менеджеру (только по своему моргу)
- Вывод:
  - Санитары (сумма расхода за период)
  - Перевозка (только для морга Мира 11)
  - Зарплата агентов (суммарно за период)
  - Доход (суммарный)
  - Вывезенные: дата, фамилия, организация (список)

### 10. Аналитика «кто вывез» (отдельная кнопка)

- Доступна: админ (оба морга), менеджер (только свой морг)
- Выбор периода: неделя / месяц / квартал
- Вывод: организация → список фамилий → количество тел
- Дополнительно: выгрузка `.csv` или `.json`

### 11. Ритуальные услуги — заказ

**Общие поля:**
- Дата заказа (авто)
- ФИО заказчика, телефон
- ФИО усопшего
- Место нахождения тела:
  - Морг Первомайская 13
  - Морг Мира 11
  - Другое (текст)
- Привязка к `shift_id`

**Похороны:**
- Где отпевают (храм / другое)
- Кладбище
- Телефон (доп., опционально)

**Кремация:**
- Дата кремации (заполняется)
- Урна:
  - Картон → в карточку «вечная память»
  - Пластик → цвет: белая / чёрная / синяя
- Допы (чекбоксы):
  - Крупное тело
  - Полированный гроб
  - Короткое прощание
  - Зал
  - Зал + отпевание
  - Срочная кремация

**Логика маршрута водителя (кремация):**
- Если выбран Зал или Зал+отпевание → конечная точка крематорий
- Иначе → кладбище

### 12. Карточки

**Водителю (похороны):**
ФИО, морг1, морг2, другой адрес, место отпевания, кладбище

**Водителю (кремация с залом):**
ФИО, морг, зал, крематорий

**Крематорию:**
```
ФИО: [ФИО]
Дата кремации: [дата]
Урна: [картон / пластик цвет]
Допы: [список]

Все стандартно, оплата наличными, оформление в день кремации.
```

### 13. Интерфейс
- Только кнопки (InlineKeyboard)
- `/start` → главное меню
- Все действия — через нажатия кнопок

### 15. Технологии и деплой (Render.com + Webhook)

**Бот должен работать на Render.com через Webhook (не polling):**

- Python 3.10+
- `aiogram 3.x`
- `uvicorn` + `FastAPI` (или `aiohttp`) для приёма webhook-запросов от Telegram
- Переменная окружения: `BOT_TOKEN`
- Webhook URL: `https://ваш-сервис.onrender.com/webhook`
- При запуске бот устанавливает webhook через `bot.set_webhook()`
- Локально для разработки — polling (через флаг `--polling`), на продакшене — webhook

**Структура запуска:**
```bash
# Продакшен (Render)
uvicorn main:app --host 0.0.0.0 --port 10000

# Локальная разработка
python bot.py --polling
```

**Требования к коду:**
- Отдельный модуль `webhook_server.py` или встроенный в `main.py`
- Поддержка `async`
- Логирование всех запросов и ошибок
- При перезапуске — повторная установка webhook

**Остальное:**
- Работа с JSON (чтение/запись без конфликтов — раздельные файлы по моргам)
- Логирование действий
- Код с комментариями, разбитый на модули:
  - `handlers/`
  - `keyboards/`
  - `database/` (JSON-прослойка)
  - `utils/` (отчёты, карточки)
  - `webhook/` (если отдельно)
